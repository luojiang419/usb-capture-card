# 016-无画面诊断与性能优化

## 已完成内容

- 读取最新快照 `015-USB拔出直接回主界面.md`，在其未提交改动上继续开发，未回退原有修改。
- 完成 AUSBC 3.2.7 依赖字节码审计，确认 `CameraActivity` 已通过 `SurfaceTextureListener` 在 Surface 就绪时打开相机。
- 创建阶段备份 `backup/015-无画面诊断与恢复前备份.md`。
- 将预览权限与录音权限解耦：
  - 实时预览只要求 `CAMERA`。
  - `RECORD_AUDIO` 只在用户开始录制时按需申请。
  - 拒绝录音权限不再导致采集画面无法显示。
- 避免已授权冷启动重复主动打开相机：
  - 已有相机权限时依赖 AUSBC 基类的 Surface 回调。
  - 权限弹窗授权后只在 Surface 已就绪且相机未打开时补开一次。
- 增加分层诊断：应用权限、USB 枚举、USB 授权、UVC 启动、首帧、断帧均有独立状态和 `UsbCapture` 日志。
- 将 `CameraStatus.START` 与真实首帧分开：只有非空 NV21 帧到达后才显示“视频帧已连接”。
- 首帧抽样计算 Y 分量平均亮度，区分“帧已到达但内容接近黑场”和“正常亮度帧”。
- 增加预览看门狗：
  - UVC 启动后 5 秒无首帧判定为首帧超时。
  - 已出画面后 3.5 秒无帧判定为断帧。
  - 每次 USB 连接最多自动重启一次，避免无限恢复循环。
  - 超过恢复额度后仍允许迟到帧重新清除错误状态。
- 性能优化：
  - 分辨率枚举从每次状态变化最多 5 次查询，缩减为 UVC 启动/分辨率错误/手动检测时单次查询。
  - 缓存分辨率列表，USB 拔出或断开时清空。
  - FPS 帧计数改用 `AtomicInteger`，时间戳使用可见性保护，修复采集线程与主线程并发读写。
- 保留并兼容上一阶段 USB 拔出后关闭菜单、退出沉浸模式并回主界面的逻辑。

```kotlin
private val previewWatchdog = object : Runnable {
    override fun run() {
        if (!previewFpsEnabled) return
        val receivedFrame = firstPreviewFrameReceived
        val referenceTime = if (receivedFrame) lastPreviewFrameAt else previewFpsStartedAt
        val timeout = if (receivedFrame) FRAME_STALL_TIMEOUT_MS else FIRST_FRAME_TIMEOUT_MS
        if (referenceTime > 0L && SystemClock.elapsedRealtime() - referenceTime >= timeout) {
            handlePreviewTimeout(receivedFrame)
            return
        }
        mainHandler.postDelayed(this, PREVIEW_WATCHDOG_INTERVAL_MS)
    }
}
```

## 软件或采集卡原因判定矩阵

- 始终没有“USB 层：设备已枚举”：优先检查 OTG、接口、线材、供电、采集卡硬件或设备过滤，不是渲染层问题。
- 显示“相机权限未授权”或“USB 设备权限未授权”：属于系统权限/应用使用流程问题。
- USB 已授权，但出现 UVC 启动失败状态码：可能是采集卡 UVC 兼容性、分辨率格式或 AUSBC/libuvc 驱动适配问题。
- UVC 已启动但 5 秒始终没有帧：优先检查 HDMI 输入、HDCP、采集卡供电、USB 带宽；也保留 libuvc 与该型号不兼容的可能。
- FPS 正常且诊断为“内容接近黑场”：软件已收到帧，优先检查信号源输出、HDCP、采集卡输入端和采集卡本身。
- FPS 正常且亮度正常，但屏幕仍看不到内容：原始帧已进入应用，优先怀疑 GLES/Surface 渲染的软件链路。
- 曾正常出帧后断流，同时出现 USB detach/disconnect：优先判断供电、线材、OTG 接触或采集卡热稳定性。
- 曾正常出帧后断流，但 USB 始终在线：软件驱动、采集卡固件和 USB 带宽均需通过更换采集卡/线材 A/B 测试继续区分。
